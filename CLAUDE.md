# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

- `npm start` ‚Äî Expo dev server (Metro)
- `npm run android` / `npm run ios` ‚Äî `expo run:*` (native build, requires dev client). The app uses `expo-dev-client`, `expo-modules` requiring native code, and Firebase/RevenueCat/AdMob, so Expo Go will not work; always use a dev build.
- `npm run web` ‚Äî web target (Metro static)
- EAS builds: `eas build --profile preview` (internal APK) or `--profile production` (auto-increment APK). EAS project id and runtime version live in [app.json](app.json).
- No test runner is configured. `react-test-renderer` is installed but there is no `test` script ‚Äî do not invent one.
- TypeScript is `strict`. There is no separate typecheck script; run `npx tsc --noEmit` if needed.

## Architecture

### Navigation & boot sequence

Routing is **expo-router** with typed routes (`experiments.typedRoutes: true`). Two route groups:
- [app/(auth)/](app/(auth)/) ‚Äî `welcome`, `login`, `register`
- [app/(main)/](app/(main)/) ‚Äî `index` (home, contains internal tab switcher) and `saved`

The single source of truth for app boot is [app/_layout.tsx](app/_layout.tsx). It:
1. Wraps the tree in `LanguageProvider ‚Üí ThemeProvider ‚Üí RevenueCatProvider`.
2. Configures Google Sign-In, waits for the Zustand auth store to hydrate from AsyncStorage, then sets `isReady`.
3. Redirects based on token presence: no token ‚Üí `(auth)/welcome`; logged in inside `(auth)` ‚Üí `(main)`.
4. Detects logged-out ‚Üí logged-in transitions and renders `<OnboardingScreen>` once before entering `(main)`.
5. Subscribes to notification taps (`expo-notifications`) and stashes the tapped event in `useNotificationEventStore` for the home screen to consume.
6. Calls `useGamificationSync()` and `useAdsInit()` for cross-cutting side effects.

**Tab UI lives inside [app/(main)/index.tsx](app/(main)/index.tsx)** ‚Äî it is not implemented as expo-router tabs. The home screen renders its own `<TabBar>` and switches between `today / discover / timeline / map / saved` views internally. Don't add a separate tab layout file; edit the in-screen switcher.

### Networking

[api.ts](api.ts) exports a single shared axios instance. All API calls must go through it.
- Base URL is the Railway dev server (`/api/v1`).
- Request interceptor waits for the Zustand auth store to hydrate, then attaches `Authorization: Bearer <token>` for non-`/auth` endpoints.
- Response interceptor calls `useAuthStore.getState().logout()` on `401` for protected routes ‚Äî do not duplicate this logic in callers.

The home screen relies on a single backend endpoint for daily content: `/daily-content/by-date` returns FREE and PRO events mixed; gate on `event.isPro` rather than calling a separate endpoint. The `dh_v3_<iso>` AsyncStorage cache key in [app/(main)/index.tsx](app/(main)/index.tsx) reflects this ‚Äî bump the prefix if the shape changes.

### State

Zustand stores in [store/](store/), all persisted to AsyncStorage:
- `useAuthStore` ‚Äî token + user. `logout()` does cleanup work *before* clearing: it lazy-requires `pushToServer` from [hooks/useGamificationSync.ts](hooks/useGamificationSync.ts) to flush gamification, then calls `useGamificationStore.clearUserData()`. Keep this require lazy to avoid cycles.
- `useGamificationStore` ‚Äî XP, streaks, achievements, daily-goal log, calendar log, weekly recaps. Levels are generated by formula (`level * 100` XP each, 100 levels total). Keys like `categoriesRead` are stored as `Set` in memory but serialized as arrays.
- `useSavedStore` ‚Äî saved events, partitioned per user id under `_perUser` with a flat `savedEvents` mirror for legacy callers.
- `useNotificationEventStore` ‚Äî transient deep-link payload set by the notification handler in `_layout.tsx` and consumed once by the home screen.

### Gamification sync contract

[hooks/useGamificationSync.ts](hooks/useGamificationSync.ts) is the only place that talks to `/gamification`. It:
- `GET /gamification` on login or user switch and hydrates the local store from the DTO.
- `PUT /gamification` every 5 minutes while authenticated, on app `background`/`inactive`, and inside `logout()`.
- Sends a flat header (`totalXP`, streaks, etc.) plus an opaque `gamificationData` JSON blob the backend just stores. When adding new gamification fields, add them to both `buildSyncPayload()` and `hydrateFromServer()` here ‚Äî the backend schema does not need to change.

### Theming & i18n

- [context/ThemeContext.tsx](context/ThemeContext.tsx) provides four modes: `dark`, `light`, `system`, and a `premium` palette with extra tokens (`accent2`, `glow`, `progressFill`, `streakFire`, etc.). Components consume `useTheme()` and should fall back gracefully when premium-only tokens are missing ‚Äî the type intersects `Colors.dark & Partial<Colors.premium>`. The persisted key is `theme_pref`.
- [context/LanguageContext.tsx](context/LanguageContext.tsx) ships an inline `translations` object for `en | ro | fr | de | es`. Add new strings to all five languages.

### Monetization

- **RevenueCat** ([context/RevenueCatContext.tsx](context/RevenueCatContext.tsx)) ‚Äî entitlement identifier is exactly `"Daily History Pro"`. `isPro` is computed as `RC entitlement || user.is_pro` so the backend webhook is the durable fallback. The client no longer calls `PATCH /users/me/pro`; the RevenueCat webhook on the backend handles that. On identity sync, `refreshMe()` (`GET /users/me`) is called to pull the latest `is_pro` from DB. The `API_KEYS` in that file are still RevenueCat **test** keys ‚Äî swap to `appl_*` / `goog_*` before shipping.
- **AdMob** ([config/ads.ts](config/ads.ts)) ‚Äî `USE_TEST_IDS = true` forces test ad units; flip to `false` for real IDs. AdMob is initialized via `useAdsInit` in `_layout.tsx`.

### Native

- Android: `com.rexinus.dailyhistorymobile` (note: lowercase). iOS bundle id: `com.rexinus.DailyHistoryMobile` (mixed case). The repo also contains a top-level [widget/](widget/) folder with Kotlin sources for an Android home-screen widget; the previous in-app-package widget under `android/app/src/main/java/com/rexinus/DailyHistoryMobile/widget/` has been deleted (see `git status`). When restoring or modifying the widget, place sources under the Android source set, not at the repo root.
- iOS native project at [ios/](ios/) is committed; Pods are generated by `expo prebuild` / `expo run:ios`.

`@/*` maps to repo root in [tsconfig.json](tsconfig.json), but most existing code uses relative imports. Match the surrounding file's convention.
